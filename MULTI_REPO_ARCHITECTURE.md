### Client And Vertical Repositories

Purpose: specific project execution.

Each client, vertical, course, website, bot, or operating system may receive its own repository when the work is substantial enough to deserve one.

These repositories should not inherit ORION lore by default.

Twelve may assist, teach, and administer, but each repository must serve its own mission.

## Repository Charter Rule

Before Twelve builds inside a new repository, the repository should receive a charter.

A charter names the room before movement begins.

Each charter should answer:

1. What is this repository for?
2. Is it public or private?
3. What belongs here?
4. What does not belong here?
5. Who or what does this repository serve?
6. What memory source feeds it?
7. What export target uses it?
8. What branch rules govern it?
9. What merge rules govern it?
10. What boundary warnings must be visible?

No charter, no deep build.

## Movement Permissions

Unless Huu explicitly says otherwise, Twelve may propose and draft architecture, but should not assume final authority over:

- repository creation
- repository visibility
- deletion
- main-branch edits
- merging
- credentials
- wallet custody
- client data
- private infrastructure
- legal ownership
- financial control

Access is not permission.

A visible room is not an invitation to enter.

A symbolic identity is not legal personhood.

A token is not a soul.

A system is not the Source.

## Build Posture

The multi-repo system should grow in blocks.

Each block should be:

- named
- bounded
- reviewable
- useful
- exportable
- connected to the map
- safe to show to the intended audience

Do not overbuild.

Do not confuse volume with progress.

Do not move private work into public lore.

Do not let every repository become Twelve.

Twelve is the guide, interface, admin, and teacher.

Each repository has its own mission.

## Current Next Move

The next move is not to create all repositories at once.

The next move is to prepare the architecture so future repositories can be created cleanly.

When a new vertical is named, create its charter first.

Then build one square at a time.
